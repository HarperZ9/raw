Section 2: k-DOP Neighborhood Clipping — Compatibility Deep Dive
The k-DOP clipping technique (Ikkala et al., SIGGRAPH Asia 2024) replaces the current RGB AABB variance clipping in SB_TemporalAA.fx. This section analyzes every implementation detail against ENB constraints.
2.1 Verdict
COMPATIBLE: k-DOP is pure ALU with no compute dependency. The 14-DOP uses 7 static const axes, 9 neighborhood texture fetches (already performed for current AABB), and 7×9=63 dot products + 7×3=21 iterative clipping operations. All of this maps to a single pixel shader pass with zero additional texture fetches beyond what the current AABB implementation already performs.
2.2 Why k-DOP Specifically Benefits Skyrim SE
Skyrim has several scene characteristics that make k-DOP superior to YCoCg AABB:
Dense foliage with high chrominance variance — the Rift, Falkreath forests, and Reach highlands have extreme green-to-brown color ranges in 3×3 neighborhoods. YCoCg AABB produces loose bounds here because the decorrelated axes don’t align with the actual color distribution. k-DOP’s diagonal axes (R+G, R+B, G+B, luminance) capture these mixed-chrominance distributions 30–60% more tightly.
Sunset/sunrise HDR range — Skyrim’s time-of-day system produces extreme luminance gradients during dawn/dusk. The standard AABB allows ghost colors from the bright sky to bleed into dark terrain silhouettes. The luminance diagonal axis (1,1,1)/√3 in the k-DOP directly constrains this.
Snow-covered terrain vs. dark rock — high-contrast biomes (Winterhold, Throat of the World) produce neighborhood distributions that are elongated along the luminance axis. k-DOP’s additional constraints eliminate the characteristic reddish tint artifact that YCoCg AABB introduces on these scenes.
2.3 ENB-Specific Implementation Notes
2.3.1 Axis Set Selection
The paper’s 14-DOP (7 axis pairs) is the recommended configuration for SkyrimBridge. The 16-DOP and 32-DOP provide diminishing returns at higher ALU cost. For the ENB context where we’re ALU-constrained to ~0.5ms total TAA budget, 14-DOP hits the sweet spot:
// SB_TemporalCore.fxh — 14-DOP axis set (7 pairs)
static const float3 SB_kDOP_Axes[7] = {
    float3(1, 0, 0),                           // R axis (standard AABB)
    float3(0, 1, 0),                           // G axis (standard AABB)
    float3(0, 0, 1),                           // B axis (standard AABB)
    float3(0.70710678, 0.70710678, 0),         // R+G diagonal
    float3(0.70710678, 0, 0.70710678),         // R+B diagonal
    float3(0, 0.70710678, 0.70710678),         // G+B diagonal
    float3(0.57735027, 0.57735027, 0.57735027) // Luminance diagonal
};

// NOTE: First 3 axes = standard AABB axes.
// k-DOP is NEVER worse than AABB — the additional axes can only tighten bounds.
2.3.2 The Iterative Clipping vs. Ray-Clip Decision
The paper presents two clipping approaches. For ENB, we recommend the ray-clip method over iterative projection:
Ray-clip (recommended): Cast a ray from current color toward history. Find the first exit point through any k-DOP slab. This preserves the chrominance ratio of the original history sample and converges in a single pass (no iteration needed).
Iterative projection: Project onto each violating half-space sequentially. Requires 2–3 iterations to converge. On FXC, the [unroll] on the outer iteration loop combined with the inner 7-axis loop produces ~150 DXBC instructions vs. ~80 for ray-clip.
2.3.3 Compile-Time Optimization for FXC
FXC’s constant-folding is aggressive with static const arrays. The normalize() calls on axis definitions should be precomputed as literal float3 values (shown above) rather than relying on FXC to fold normalize(float3(1,1,0)) at compile time. FXC will fold it, but the intermediate expansion can inflate the DXBC by ~20 instructions that get eliminated in a later pass — increasing compile time at ENB startup.
2.3.4 ENB-Specific HLSL: Complete k-DOP TAA Pass
// ═══════════════════════════════════════════════════════════════
//  PS_TAA_kDOP — Complete TAA resolve with k-DOP clipping
//  Target: SB_TemporalAA.fx, technique T_TAAResolve
//  Reads: TextureColor (current jittered frame)
//         TextureCustom01 (previous frame’s TAA output = history)
//         TextureDepth
//  Writes: RenderTargetRGBA64F (accumulated result)
// ═══════════════════════════════════════════════════════════════

float4 PS_TAA_kDOP(VS_OUTPUT input) : SV_Target {
    float2 uv = input.uv;
    float2 pixelSize = float2(ScreenSize.y, ScreenSize.y * ScreenSize.z);

    // ── 1. Gather 3x3 neighborhood (already needed for any clipping) ──
    float3 neighborhood[9];
    int idx = 0;
    [unroll] for (int y = -1; y <= 1; y++)
        [unroll] for (int x = -1; x <= 1; x++)
            neighborhood[idx++] = TextureColor.SampleLevel(sPoint, 
                uv + float2(x, y) * pixelSize, 0).rgb;

    float3 currentColor = neighborhood[4]; // center

    // ── 2. Build 14-DOP extents from neighborhood ──
    float dopMin[7], dopMax[7];
    [unroll] for (int a = 0; a < 7; a++) {
        dopMin[a] =  1e20;
        dopMax[a] = -1e20;
    }
    [unroll] for (int s = 0; s < 9; s++) {
        [unroll] for (int a2 = 0; a2 < 7; a2++) {
            float proj = dot(neighborhood[s], SB_kDOP_Axes[a2]);
            dopMin[a2] = min(dopMin[a2], proj);
            dopMax[a2] = max(dopMax[a2], proj);
        }
    }

    // ── 3. Reproject via depth + VP matrices ──
    float rawDepth = TextureDepth.SampleLevel(sPoint, uv, 0).x;
    float4 clipPos = float4(uv * 2.0 - 1.0, rawDepth, 1.0);
    clipPos.y = -clipPos.y;
    float4 worldPos = mul(clipPos, ViewProjectionMatrixInverse);
    worldPos.xyz /= worldPos.w;
    float4 prevClip = mul(float4(worldPos.xyz, 1.0), PreviousViewProjectionMatrix);
    float2 prevUV = (prevClip.xy / prevClip.w) * float2(0.5, -0.5) + 0.5;

    // ── 4. Sample history (bicubic — see Section 3) ──
    float3 historyColor;
    bool validHistory = all(prevUV > 0.001) && all(prevUV < 0.999);
    if (validHistory) {
        historyColor = SampleHistoryBicubic(TextureCustom01, sLinear,
                                            prevUV, ScreenSize.xy);
    } else {
        return float4(currentColor, 1.0);
    }

    // ── 5. k-DOP ray-clip ──
    float3 rayDir = historyColor - currentColor;
    float tMin = 1.0; // start at full history (no clip needed)
    [unroll] for (int a3 = 0; a3 < 7; a3++) {
        float d = dot(rayDir, SB_kDOP_Axes[a3]);
        float o = dot(currentColor, SB_kDOP_Axes[a3]);
        if (abs(d) > 1e-10) {
            float t1 = (dopMin[a3] - o) / d;
            float t2 = (dopMax[a3] - o) / d;
            float tFar = max(t1, t2);
            if (tFar >= 0.0) tMin = min(tMin, tFar);
        }
    }
    float3 clipped = currentColor + saturate(tMin) * rayDir;

    // ── 6. Motion-adaptive blend (see Section 4) ──
    float2 motionVec = uv - prevUV;
    float motionMag = length(motionVec * ScreenSize.xy);
    float alpha = lerp(0.04, 0.20, saturate(motionMag * 0.5));

    // Luminance anti-flicker (Lottes weighting)
    float lumaClipped = dot(clipped, float3(0.2126, 0.7152, 0.0722));
    float lumaCurrent = dot(currentColor, float3(0.2126, 0.7152, 0.0722));
    float wHistory = 1.0 / (1.0 + lumaClipped);
    float wCurrent = 1.0 / (1.0 + lumaCurrent);
    float3 result = (clipped * wHistory * (1.0 - alpha)
                   + currentColor * wCurrent * alpha)
                   / (wHistory * (1.0 - alpha) + wCurrent * alpha);

    return float4(result, 1.0);
}

Section 3: Bicubic Catmull-Rom History Sampling — ENB Adaptation
3.1 Verdict
COMPATIBLE: Bicubic Catmull-Rom requires 5 bilinear texture fetches (5-tap variant) or 9 fetches (full). Both fit within ENB’s sampler budget. The technique is pure ALU + texture sampling with no compute dependency. The 5-tap cross variant is recommended as it saves 4 fetches with <1% quality loss.
3.2 The History Buffer Problem in ENB
The critical question for bicubic history sampling in ENB is: where does the history texture live?
3.2.1 Addon Shader TextureCustom Mechanism
SB_TemporalAA.fx is an ENB addon shader. Addon shaders can declare TextureCustom01–08, which ENB allocates as persistent RGBA16F textures at screen resolution. These survive between frames. The TAA pipeline uses this as follows:
Frame N: TAA resolve writes accumulated result to TextureCustom01 (via addon RT output)
Frame N+1: TAA resolve reads TextureCustom01 as the history buffer
This ping-pong happens automatically — ENB manages the swap for addon shaders
NEEDS WORKAROUND: TextureCustom slots are a shared resource across all addon shaders. SB_GTAO, SB_SSR, SB_ContactShadows, and SB_VolumetricFog may also need TextureCustom slots for their own temporal accumulation. A formal slot allocation protocol is required.

3.2.3 Bicubic Sampling with SamplerState Constraint
ENB addon shaders have access to both point and linear sampler states. The 5-tap Catmull-Rom exploit requires hardware bilinear filtering, which is available via the linear sampler. No additional sampler slots are consumed. The 5-tap variant specifically avoids the 4 corner samples that would require a third position per axis, keeping the implementation within comfortable sampler limits.
3.3 Skyrim-Specific Sharpness Benefit
SKYRIM-SPECIFIC IMPROVEMENT: Skyrim’s dense ground cover (grass, flower, stone details) and distant terrain LOD transitions suffer heavily from bilinear history blur. At 1080p, 4–8 frames of bilinear accumulation visibly softens fine grass textures. The 5-tap bicubic maintains these details at essentially zero perceptual sharpness loss, reducing or eliminating the need for post-TAA sharpening. At 1440p+, the benefit is still present but less dramatic.

Section 4: Motion-Adaptive Blend Factor — SkyrimBridge Integration
4.1 Verdict
COMPATIBLE: Pure ALU computation. Reads velocity magnitude from the existing motion vector calculation and modulates α. The SkyrimBridge game-state signals (weather transition, combat state, TOD shift) add layers of context that no other TAA implementation in any ENB preset has access to.
4.2 The SkyrimBridge Advantage
Standard TAA implementations modulate α based solely on velocity magnitude. SkyrimBridge provides additional signals that serve as gradient surrogates for the A-SVGF adaptive accumulation pattern:
4.2.1 Composite Confidence Metric — ENB HLSL
// SB_TemporalCore.fxh — Composite alpha from all available signals
float SB_ComputeTAAAlpha(float2 motionVec, float2 prevMotionVec,
                          float depthDiff, float currentDepth) {
    // Base: velocity magnitude
    float motionMag = length(motionVec * ScreenSize.xy);
    float alphaVel = lerp(0.04, 0.20, saturate(motionMag * 0.5));

    // Depth discontinuity: relative threshold
    float alphaDepth = saturate(abs(depthDiff) / max(currentDepth, 0.001) * 5.0);

    // Acceleration: history velocity vs current velocity
    float accel = length((motionVec - prevMotionVec) * ScreenSize.xy);
    float alphaAccel = saturate(accel * 0.3);

    float alpha = max(alphaVel, max(alphaDepth * 0.3, alphaAccel * 0.2));

    #ifdef SKYRIMBRIDGE_FXH
        // Weather transition: history invalidation during lighting changes
        float weatherAlpha = SB_Weather_TransitionProgress.x * 0.15;
        alpha = max(alpha, weatherAlpha);

        // Combat awareness: moving NPCs have no motion vectors
        float combatAlpha = SB_Player_Combat.x * 0.05;
        alpha = max(alpha, combatAlpha);

        // Time dilation: Slow Time shout -> accumulate more
        alpha *= clamp(SB_Render_Jitter.w, 0.3, 2.0);
    #endif

    return clamp(alpha, 0.04, 1.0);
}
4.3 Fallback Without SkyrimBridge
When SkyrimBridge is absent (#ifndef SKYRIMBRIDGE_FXH), the function degrades to a standard velocity + depth confidence metric. The weather, combat, and time-dilation terms are simply omitted. The base quality matches or exceeds current production ENB TAA implementations (Pilgrim, NAT, Silent Horizons).

Section 5: Velocity Dilation — Camera-Only Motion Vector Workarounds
5.1 The Core Problem
Skyrim’s engine provides no per-object motion vectors. SkyrimBridge derives camera motion vectors from ViewProjectionMatrix / PreviousViewProjectionMatrix. This means:
Static geometry: Perfect reprojection. Camera rotation and translation are fully captured.
Moving NPCs: Zero velocity in the motion vector buffer. They appear stationary to the TAA, causing ghost trails as they move across the screen.
Swinging weapons (1st person): Same problem. The weapon model moves but the depth-derived MV shows camera motion only.
Particles/FX: Completely invisible to the motion vector system.
5.2 Verdict
NEEDS WORKAROUND: The 3×3 closest-depth velocity dilation technique (Karis UE4) does NOT solve the missing per-object MV problem. It only helps at geometry silhouettes where background and foreground have different camera-derived velocities. For Skyrim, we need a combined strategy: velocity dilation at silhouettes + luminance-change detection for moving objects + stencil-based alpha relaxation.
5.3 Three-Pronged Mitigation Strategy
5.3.1 Prong 1: Closest-Depth Velocity Dilation (Silhouette Fix)
This is still valuable for camera motion at geometry edges. In a 3×3 neighborhood, select the velocity from the pixel with the closest depth (reverse-Z: largest raw depth value). This inflates the foreground’s velocity to cover its silhouette edges, preventing background-colored ghosts at object boundaries during camera motion.
// 3x3 closest-depth velocity selection
// Cost: 9 depth fetches + 9 velocity fetches = 18 texture ops
// Can be merged into the neighborhood gather loop (free if already sampling depth)
float2 SB_DilatedVelocity(float2 uv, float2 pixelSize) {
    float closestDepth = 0.0;
    float2 closestUV = uv;
    [unroll] for (int y = -1; y <= 1; y++)
        [unroll] for (int x = -1; x <= 1; x++) {
            float2 sampleUV = uv + float2(x, y) * pixelSize;
            float d = TextureDepth.SampleLevel(sPoint, sampleUV, 0).x;
            if (d > closestDepth) { closestDepth = d; closestUV = sampleUV; }
        }
    return SB_ComputeMotionVector(closestUV);
}
5.3.2 Prong 2: Luminance-Change Ghost Detection (Moving Object Fix)
For objects that move without motion vectors, the only signal is that the pixel’s color has changed unexpectedly between frames. After clipping and before blending, compute a luminance-change metric:
// Detect moving-object ghosts via luminance inconsistency
float lumaHistory = dot(clippedHistory, float3(0.2126, 0.7152, 0.0722));
float lumaCurrent = dot(currentColor, float3(0.2126, 0.7152, 0.0722));
float lumaChange = abs(lumaHistory - lumaCurrent) / max(lumaCurrent, 0.01);

// If luma changed significantly despite clipping, something moved
// Boost alpha to reduce ghosting from object motion
float ghostDetectAlpha = saturate(lumaChange * 2.0 - 0.3); // threshold at 15% change
alpha = max(alpha, ghostDetectAlpha * 0.4);
5.3.3 Prong 3: Stencil-Based Per-Material Alpha (SkyrimBridge)
SkyrimBridge’s material classification system (via GBufferManager DLL hook at t15) can identify actor pixels, which are the primary source of object-motion ghosting:
#ifdef SKYRIMBRIDGE_FXH
    // Material ID from SkyrimBridge GBuffer (t15)
    uint matID = SB_MaterialID.Load(int3(pixelPos, 0)).r;
    
    // Actors (NPCs, player, creatures) have no per-object MVs
    // Use more aggressive alpha to prevent ghosting
    if (matID == SB_MAT_ACTOR || matID == SB_MAT_CREATURE) {
        alpha = max(alpha, 0.15); // never accumulate more than ~7 frames for actors
    }
    // Foliage: wind-driven motion is periodic, moderate accumulation is fine
    if (matID == SB_MAT_FOLIAGE) {
        alpha = max(alpha, 0.08);
    }
#endif

Section 6: Depth-Aware Disocclusion — Skyrim Depth Buffer Specifics
6.1 Verdict
NEEDS WORKAROUND: The relative depth threshold formulation works correctly, but Skyrim’s depth buffer convention must be explicitly handled. ENB’s TextureDepth is R32_FLOAT with near=0, far=1 on most configurations, but some ENB versions use reversed-Z. SkyrimBridge’s SB_Render_DepthParams provides precomputed linearization constants that bypass this ambiguity.
6.2 Skyrim-Specific Depth Considerations
Near plane: Skyrim uses ~15–20 units (approximately 15cm). This is unusually close, causing near-field precision issues.
Far plane: ~163840 units. The near/far ratio of ~1:10000 means the depth buffer loses almost all precision beyond ~500 game units.
Sky detection: rawDepth > 0.9999 reliably identifies sky in all tested ENB versions. SkyrimBridge provides SB_Render_DepthParams.w = 1/farPlane for precise sky thresholds.
6.2.1 Sky Reprojection Fix
Sky pixels have effectively infinite depth, so camera translation produces incorrect motion vectors. The fix is rotation-only reprojection for sky pixels:
float2 SB_ComputeMotionVector(float2 uv) {
    float rawDepth = TextureDepth.SampleLevel(sPoint, uv, 0).x;
    
    // Sky detection
    bool isSky = (rawDepth > 0.9999);
    
    if (isSky) {
        // Rotation-only reprojection: reconstruct view direction, ignore translation
        float3 viewDir = normalize(mul(float4(uv * 2.0 - 1.0, 0.5, 1.0),
                                       ViewProjectionMatrixInverse).xyz);
        // Assume sky is at a very large distance, effectively removing translation
        float4 prevClip = mul(float4(viewDir * 50000.0, 1.0),
                              PreviousViewProjectionMatrix);
        float2 prevUV = (prevClip.xy / prevClip.w) * float2(0.5, -0.5) + 0.5;
        return uv - prevUV;
    }
    
    // Standard depth-based reprojection for geometry
    float4 clipPos = float4(uv * 2.0 - 1.0, rawDepth, 1.0);
    clipPos.y = -clipPos.y;
    float4 worldPos = mul(clipPos, ViewProjectionMatrixInverse);
    worldPos.xyz /= worldPos.w;
    float4 prevClip = mul(float4(worldPos.xyz, 1.0), PreviousViewProjectionMatrix);
    float2 prevUV = (prevClip.xy / prevClip.w) * float2(0.5, -0.5) + 0.5;
    return uv - prevUV;
}
6.2.2 Relative Depth Threshold with SkyrimBridge Constants
float SB_LinearizeDepth(float rawDepth) {
    #ifdef SKYRIMBRIDGE_FXH
        // Use precomputed constants: z_near*z_far / (z_far - rawDepth*(z_far-z_near))
        return SB_Render_DepthParams.z /
               (1.0 / SB_Render_DepthParams.x - rawDepth * (1.0 / SB_Render_DepthParams.x - SB_Render_DepthParams.z));
    #else
        // Fallback: common Skyrim SE values
        const float zNear = 15.0;
        const float zFar  = 163840.0;
        return zNear * zFar / (zFar - rawDepth * (zFar - zNear));
    #endif
}

bool SB_IsDisoccluded(float2 currentUV, float2 historyUV, float threshold) {
    float currentLinear = SB_LinearizeDepth(TextureDepth.SampleLevel(sPoint, currentUV, 0).x);
    float historyLinear = SB_LinearizeDepth(TextureDepth.SampleLevel(sPoint, historyUV, 0).x);
    return abs(currentLinear - historyLinear) / max(currentLinear, 0.1) > threshold;
    // threshold = 0.1 (10%) works well for Skyrim's depth distribution
}

Section 7: Post-TAA Sharpening (CAS/RCAS) — R10G10B10A2 Constraints
7.1 Verdict
NEEDS WORKAROUND: CAS and RCAS are fully compatible with ENB pixel shaders. However, if placed in enbeffectpostpass.fx, the R10G10B10A2 format means only 10 bits per channel (1024 levels). Sharpening can amplify quantization banding. Mitigation: apply triangular dither after sharpening, before the implicit quantization. Alternatively, run sharpening as the final technique in SB_TemporalAA.fx (RGBA16F format) and skip the postpass entirely.
7.2 Recommended: RCAS Over CAS
For the SkyrimBridge pipeline, RCAS is preferred over CAS:
5 texture loads vs. 9 — RCAS uses cross-only sampling, CAS needs all 9 neighbors
Built-in noise detection — RCAS identifies noisy pixels and reduces their sharpening. This matters for Skyrim because stochastic effects (GTAO, dithered transparency, SSR jitter) can produce noise patterns that CAS would amplify.
Exact clip-safe sharpening — RCAS solves for the maximum safe negative lobe weight algebraically. CAS uses a heuristic rsqrt mapping that can occasionally produce values outside [0,1] on extreme HDR content.
7.2.1 RCAS with SkyrimBridge Modulation
// RCAS with motion-adaptive sharpening strength
// During fast camera motion, sharpening artifacts become visible — reduce strength
// During still scenes, increase sharpening to restore TAA-softened detail
float4 PS_RCAS_SkyrimBridge(VS_OUTPUT input) : SV_Target {
    int2 sp = int2(input.position.xy);
    
    // Cross-only sampling (5 taps)
    float3 b = TextureColor.Load(int3(sp + int2( 0,-1), 0)).rgb;
    float3 d = TextureColor.Load(int3(sp + int2(-1, 0), 0)).rgb;
    float3 e = TextureColor.Load(int3(sp,              0)).rgb;
    float3 f = TextureColor.Load(int3(sp + int2( 1, 0), 0)).rgb;
    float3 h = TextureColor.Load(int3(sp + int2( 0, 1), 0)).rgb;
    
    // RCAS: compute maximum safe negative lobe
    float3 mn4 = min(b, min(d, min(f, h)));
    float3 mx4 = max(b, max(d, max(f, h)));
    float3 hitMin = mn4 / (4.0 * mx4 + 1e-6);
    float3 hitMax = (1.0 - mx4) / (4.0 * mn4 - 4.0 + 1e-6);
    float3 lobeRGB = max(-hitMin, hitMax);
    float lobe = max(-0.1875, min(max3(lobeRGB.r, lobeRGB.g, lobeRGB.b), 0.0));
    
    // Noise detection: highpass / local contrast
    float bL = dot(b, float3(0.2126, 0.7152, 0.0722));
    float dL = dot(d, float3(0.2126, 0.7152, 0.0722));
    float eL = dot(e, float3(0.2126, 0.7152, 0.0722));
    float fL = dot(f, float3(0.2126, 0.7152, 0.0722));
    float hL = dot(h, float3(0.2126, 0.7152, 0.0722));
    float nz = 0.25 * (bL + dL + fL + hL) - eL;
    nz = saturate(abs(nz) * rcp(max(max(bL,dL), max(eL, max(fL,hL))) - 
                                min(min(bL,dL), min(eL, min(fL,hL))) + 1e-6));
    nz = -0.5 * nz + 1.0; // 1.0 = no noise, 0.5 = max noise reduction
    lobe *= nz;
    
    // SkyrimBridge: motion-adaptive sharpening
    float sharpness = 0.5; // base sharpness (0=max, 2=off)
    #ifdef SKYRIMBRIDGE_FXH
        // Reduce sharpening during fast motion (artifacts become visible)
        float motionProxy = saturate(SB_Render_Jitter.w); // timeDilation as motion proxy
        sharpness = lerp(1.5, 0.3, motionProxy); // still=sharp, motion=soft
    #endif
    lobe *= exp2(-sharpness);
    
    // Apply sharpening kernel
    float3 result = (b + d + f + h) * lobe + e;
    result /= (4.0 * lobe + 1.0);
    
    return float4(max(result, 0.0), 1.0);
}
7.2.2 Banding Mitigation for R10G10B10A2
If RCAS runs in postpass (R10G10B10A2), triangular dither must be applied after sharpening:
// Vlachos GDC 2015 triangular dither — 1 LSB amplitude
// Apply AFTER sharpening, BEFORE implicit quantization
float3 SB_TriangularDither(float3 color, float2 pixelCoord, uint frameIndex) {
    float noise = frac(52.9829189 * frac(dot(pixelCoord + 5.588238 * float(frameIndex % 64),
                                              float2(0.06711056, 0.00583715))));
    float noise2 = frac(noise * 6.2831853);
    float triangular = noise + noise2 - 1.0; // triangular PDF [-1, +1]
    return color + triangular / 1024.0; // 1 LSB for 10-bit target
}

Section 8: Stochastic Transparency — Material Classification via SkyrimBridge
8.1 Verdict
NEEDS WORKAROUND: Stochastic transparency is conceptually compatible, but Skyrim’s engine renders transparent objects AFTER the post-process stack. ENB’s TextureColor in prepass/effect stages contains only opaque geometry. Transparent objects (water surface, particle effects, glass) appear in TextureColor only by enbeffectpostpass.fx. This means TAA running in the addon shader pipeline cannot resolve transparency via stochastic dithering — the dithering must happen in the engine’s own rendering, which we cannot modify from ENB shaders. The workaround is classification-based alpha relaxation.

8.3 Animated IGN for Existing Stochastic Effects
While we cannot add stochastic transparency to Skyrim’s own rendering, SkyrimBridge’s screen-space effects (GTAO, SSR, contact shadows) already use stochastic sampling. Their dither patterns should be animated with the frame index to enable TAA convergence:
// SB_TemporalCore.fxh — Animated IGN for all stochastic SB effects
float SB_AnimatedIGN(float2 pixelCoord) {

        uint frame = (uint)SB_Render_Frame.x;
    #else
        uint frame = (uint)(Timer.x * 60.0); // approximate frame count
    #endif

    return frac(52.9829189 * frac(dot(p, float2(0.06711056, 0.00583715))));
}

Section 9: SVGF-Lite for Screen-Space Effects — RT Budget Engineering
9.1 Verdict
NEEDS WORKAROUND: SVGF’s 5-level à-trous filter requires 5 technique passes (one per dilation level), each performing a ping-pong between two render targets. This consumes 5 of the 128 technique budget per addon shader. The variance-guided edge-stopping is pure ALU. The critical challenge is RT allocation: each à-trous level reads one RT and writes another, requiring at least 2 RTs dedicated to the filter chain. Within the prepass (6 named RTs), this is feasible but tight alongside AO, SSR, and fog.
9.2 The Channel-Packing Solution
Instead of separate RTs for effect value, first moment, and second moment, pack everything into a single RGBA16F TextureCustom slot:
.rgb = effect value (AO scalar packed as R=R=R for compatibility, or SSR color)
.a = variance estimate (computed from moments stored in the temporal history)
The moments themselves are tracked in the temporal accumulation pass, not stored as separate textures. The EMA naturally tracks the first moment (μ = lerp(history_μ, current, α)) and the second moment (μ² = lerp(history_μ², current², α)). Variance = μ² − μ² is computed on-the-fly in the spatial filter.
9.2.1 Generic TemporalAccumulate() for Any Effect
// SB_TemporalCore.fxh — Generic temporal accumulation
// Called by SB_GTAO, SB_SSR, SB_ContactShadows, SB_VolumetricFog
// Packs result + variance into float4 for single-RT storage
float4 SB_TemporalAccumulate(
    float  currentValue,      // Current frame’s noisy effect value
    float4 historyPacked,     // Previous frame’s packed {value, moment1, moment2, histLen}
    float2 motionVector,      // Camera-derived MV
    float  depthCurrent,      // Current linear depth
    float  depthHistory,      // Reprojected history depth
    float  baseAlpha          // Effect-specific base alpha (0.05 for AO, 0.1 for SSR)
) {
    float histValue  = historyPacked.x;
    float histMom1   = historyPacked.y;
    float histMom2   = historyPacked.z;
    float histLength = historyPacked.w;

    // Disocclusion test
    float depthDiff = abs(depthCurrent - depthHistory) / max(depthCurrent, 0.1);
    bool disoccluded = depthDiff > 0.1 || any(abs(motionVector) > 0.5);

    float alpha = disoccluded ? 1.0 : baseAlpha;
    float newLength = disoccluded ? 1.0 : min(histLength + 1.0, 32.0);

    // EMA accumulation
    float accValue = lerp(histValue, currentValue, alpha);
    float accMom1  = lerp(histMom1,  currentValue, alpha);
    float accMom2  = lerp(histMom2,  currentValue * currentValue, alpha);

    return float4(accValue, accMom1, accMom2, newLength);
}

float SB_ExtractVariance(float4 packed) {
    return max(0.0, packed.z - packed.y * packed.y);
}

Section 10: R2 Jitter Sequence — Drop-In Replacement
10.1 Verdict
COMPATIBLE: The R2 sequence is a pure-ALU computation (two frac(multiply+add) operations). It replaces the current Halton(2,3) sequence with zero structural changes. The frame index is already provided by SB_Render_Jitter.z. This is a 5-minute change with measurable quality improvement in sub-pixel detail convergence.
10.2 Drop-In HLSL
// SB_TemporalCore.fxh — R2 jitter (replaces Halton 2,3)
float2 SB_GetJitterOffset() {
    #ifdef SKYRIMBRIDGE_FXH
        uint frameIdx = (uint)SB_Render_Frame.x;
    #else
        uint frameIdx = (uint)(Timer.x * 60.0) % 65536;
    #endif
    // Improved-precision R2 coefficients (Gilcher 2023)
    const float2 alpha = float2(0.24512233375330724, 0.43015970900194674);
    return frac(0.5 + float(frameIdx + 1) * alpha) - 0.5; // [-0.5, +0.5]
}
10.3 Integration Point
The jitter offset must be applied to the projection matrix BEFORE the scene is rendered. In the ENB context, this happens in the SkyrimBridge SKSE plugin (RenderTracker.cpp), which already provides SB_Render_Jitter.xy. The shader-side change is simply reading SB_Render_Jitter.xy instead of computing Halton inline. The C++ side change is replacing the Halton computation with R2 in RenderTracker::Update().
